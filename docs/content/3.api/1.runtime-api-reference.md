## Module options

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['nuxt-google-sheets-import'],
  googleSheetsImport: {
    apiBase: '/api/google-sheets-import',
    defaultContentDir: 'content/data',
    googleSheets: [
      { id: '<SPREADSHEET_ID_1>', label: 'Primary Sheet' },
      { id: '<SPREADSHEET_ID_2>', label: 'Staging Sheet' },
    ],
  }
})
```

- `apiBase`: base route for module API handlers
- `defaultContentDir`: default write target for imported files
- `googleSheets`: source spreadsheet options shown in the import tab

Google API keys are configured in Nuxt runtime config (for example `runtimeConfig.googleApiKey`) and consumed by the server routes.

## Built-in route

The module registers a page at:

```text
/google-sheets-import
```

This page provides three tabs:

- `Setup Google Sheet`
- `Import data`

## API endpoints

Base path: `{apiBase}` (default `/api/google-sheets-import`)

- `GET /sheets`
  - Query: `spreadsheetId`
  - Returns available sheet tabs.
- `POST /values`
  - Validates and transforms sheet values against a selected schema.
- `GET /collection-type`
  - Resolves whether a schema maps to a `page` or `data` collection.
- `GET /schema-columns`
  - Query: `schema?`
  - Returns:
    - `schemas`
    - `columns`
    - `collectionType`
    - `pageOverrideColumns`
- `POST /write`
  - Writes transformed records to markdown frontmatter, JSON, or YAML.

## Export behavior details

- Clipboard copy in export uses TSV for reliable Google Sheets paste.
- File download uses quoted CSV for file-based data exchange.

## Write behavior

Supported overwrite strategies:

- `Skip existing files` (`skip`)
- `Overwrite frontmatter only (.md files)` (`overwrite-frontmatter`)
- `Overwrite all` (`overwrite`)

Definitions:

- `skip`: if the target file already exists, no write is performed for that record.
- `overwrite-frontmatter`: for Markdown output, frontmatter is replaced while existing markdown body content is preserved.
- `overwrite`: replaces the full target file content when it already exists.

## Suggested cell value patterns

### String

```ts [zod schema example]
z.string()
```

- Column header: `title`
- Cell contents: `Example text`
- Notes: trimmed text is passed through as a string.

### Number

```ts [zod schema example]
z.number()
```

- Column header: `price`
- Cell contents: `123`
- Notes: parsed with JavaScript `Number(...)` when possible; invalid numbers stay strings and then fail Zod validation.

### Boolean

```ts [zod schema example]
z.boolean()
```

- Column header: `isPublished`
- Cell contents: `false`, `FALSE`, `0`, `no` | `true`, `TRUE`, `1`, `yes`

### Enum

```ts [zod schema example]
z.enum(['draft', 'published'])
```

- Column header: `status`
- Cell contents: `draft`
- Notes: no enum coercion is applied; value must match one of the allowed enum values.

### Date-like String

```ts [zod schema example]
z.string().datetime()
  .or(z.string().regex(/^\d{4}-\d{2}-\d{2}$/))
```

- Column header: `publishedAt`
- Cell contents: `2026-01-01`
- Notes: no date coercion is applied in the transform step; it remains a string and is validated by Zod.

### String Array

```ts [zod schema example]
z.array(z.string())
```

- Column header: `tags`
- Cell contents: `foo, bar, baz`
- Notes: split on commas, trim each item, and remove empty entries. Empty cell becomes `[]` (or `undefined`/`null` for optional/nullable).

### Object Array

```ts [zod schema example]
z.array(
  z.object({ 
    name: z.string(), 
    price: z.number() 
  })
)
```

- Column header: `items[0].name`, `items[0].price`
- Cell contents: `Starter Pack`, `49`
- Notes: use indexed headers for object arrays (`items[1].*`, etc.); nested scalar fields still use their own coercion rules.
